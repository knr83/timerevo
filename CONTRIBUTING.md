# Contributing to Timerevo

Thanks for your interest in contributing to Timerevo.

Timerevo is a privacy-first desktop time tracking application for small teams. The project aims to stay simple, practical, and maintainable. Please read these guidelines before opening an issue or submitting a pull request.

## First steps

Before contributing, please:

- read the `README.md`
- review the project documentation in `docs/`
- check existing issues and discussions
- make sure your idea fits the current scope of the project

For non-trivial changes, please open an issue first so the approach can be discussed before implementation starts.

## What is welcome

Contributions are especially welcome for:

- bug fixes
- usability improvements
- performance improvements
- code cleanup with clear value
- tests
- documentation updates
- small features that fit the existing product direction

## What is usually not accepted

The following types of contributions are less likely to be accepted unless discussed in advance:

- large architectural refactors
- changes that add unnecessary complexity
- new dependencies without strong justification
- features outside the current product scope
- opinionated rewrites of existing working code

## Contribution principles

Please keep contributions aligned with these principles:

- prefer simple solutions
- keep changes focused
- respect existing architecture and boundaries
- avoid overengineering
- optimize for maintainability
- keep product behavior predictable and consistent

## Code expectations

Please follow the current project style and structure.

General expectations:

- write clear and readable code
- use meaningful naming
- avoid unrelated changes in the same pull request
- keep abstractions justified
- do not leave unused code behind
- keep comments minimal and useful

## Documentation

If your change affects project behavior, development workflow, architecture, or setup, update the relevant documentation in `docs/` in the same pull request.

Documentation should reflect the actual state of the project.

## Reporting bugs

When opening a bug report, please include:

- what happened
- how to reproduce it
- what you expected instead
- environment details if relevant
- screenshots, logs, or examples if useful

## Suggesting features

When suggesting a feature, please describe:

- the problem it solves
- who benefits from it
- the expected behavior
- why it fits Timerevo

## Pull request guidelines

Please keep pull requests small, focused, and reviewable.

Before submitting a pull request, make sure that:

- the project builds successfully
- the change works as intended
- relevant documentation was updated if needed
- no unrelated files are included by mistake

A pull request should clearly explain:

- what changed
- why it changed
- whether there are any trade-offs or limitations

## Final note

Timerevo is developed with a clear product and architecture direction. Not every technically valid contribution will be merged.

If you are unsure whether your contribution is a good fit, open an issue first and discuss the idea before implementing it.

Thanks for contributing.
